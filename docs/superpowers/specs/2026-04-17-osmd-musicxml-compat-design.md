# OSMD MusicXML Compatibility Fixes

**Date:** 2026-04-17
**Status:** Draft
**Branch:** feat/GAU-105-llm-engrave-superpowers

## Problem

The MusicXML generated by the engrave stage renders correctly through the LilyPond PDF pipeline but displays incorrectly in OSMD (OpenSheetMusicDisplay v1.8.9) in the Flutter frontend:

1. **UUID instrument names** — OSMD shows "Instr. P694f98b296fdb9b8d07493d844452ab1" instead of "Piano" because it reads `<part-name>` from `<score-part>`, not `<group-name>` from `<part-group>`. music21 populates `<part-name>` with its internal part ID when no explicit `partName` is set.
2. **Note collisions** — Notes overlap and pile up in certain measures, likely due to duration/division mismatches, missing `<chord/>` tags, or voice ambiguity that OSMD handles differently than LilyPond.
3. **No isolated test path** — Debugging requires running the full pipeline end-to-end.

## Approach

Extend `_sanitize_musicxml_for_osmd()` in `backend/services/engrave.py` with additional post-processing passes, and add a test harness for fast iteration.

## Design

### Fix 1: Strip UUID Part Names

Add a pass in `_sanitize_musicxml_for_osmd()` that rewrites `<part-name>` elements in `<score-part>` entries:

- Set `<part-name>` text to empty string (`""`) for all parts
- The `<part-group>` wrapper already carries `<group-name>Piano</group-name>` from the StaffGroup — OSMD uses this for the brace label
- This matches the intent of the existing code comment: "per-part names would render twice (once per staff) in some tools"

**Target:** `_sanitize_musicxml_for_osmd()` in `backend/services/engrave.py`

### Fix 2: Diagnose and Fix Note Collisions

Investigate the specific MusicXML patterns causing OSMD rendering collisions. Likely candidates:

- Notes whose durations don't fit the `divisions=12` grid, causing OSMD to place them at the same x-position
- Simultaneous notes missing `<chord/>` tags — should share a stem but get separate voice treatment
- Voice assignment edge cases not caught by the existing `_remap_voices_per_staff()` pass

**Process:**
1. Capture a problematic MusicXML file from the current pipeline output
2. Inspect the XML around garbled measures to identify the pattern
3. Add a targeted sanitizer pass to fix it
4. Verify in the OSMD test harness (Fix 3)

**Target:** New pass(es) in `_sanitize_musicxml_for_osmd()`, pattern TBD after diagnosis

### Fix 3: OSMD Test Harness

Add tooling for fast isolated iteration without running the full pipeline:

**A. Static OSMD viewer page** — `tests/osmd_viewer.html`
- Loads OSMD 1.8.9 from CDN (same version as Flutter app)
- Mirrors the Flutter OSMD config: SVG backend, `drawPartNames: true`, `drawPartAbbreviations: false`, `drawMeasureNumbers: true`, cursor enabled
- File input or drag-and-drop to load a local `.musicxml` file
- Open in any browser for visual inspection

**B. MusicXML extraction command** — pytest fixture or standalone script
- Takes a PianoScore JSON fixture and runs only `_render_musicxml_bytes()` + `_sanitize_musicxml_for_osmd()`
- Writes the output `.musicxml` to a known location
- Skips MIDI rendering, PDF rendering, blob storage, and all other pipeline stages

**C. Regression fixture** — Save the "Jammin'" MusicXML as a test fixture (`tests/fixtures/jammin_osmd_regression.musicxml`) to verify fixes don't regress.
